Sound Processor — конспект для защиты
=====================================

Этот файл нужен для подготовки к code review и устной защите. В нем описано, что делает каждый файл проекта, какие принципы C++ и архитектуры использованы, и как объяснить основные решения по ТЗ.

1. Общая идея проекта
---------------------

Проект — консольное CLI-приложение для обработки WAV-файлов.

Пользователь запускает программу так:

    sound_processor -i input.wav -o output.wav -f ampl 0.8 -f lowpass 51

Приложение:
1. Разбирает аргументы командной строки.
2. По описанию фильтров строит Pipeline.
3. Читает входной WAV в Waveform.
4. Последовательно применяет фильтры.
5. Записывает результат в выходной WAV.

Важно: основная логика обработки звука не зависит от того, откуда пришли параметры. Сейчас параметры приходят из командной строки, но архитектура позволяет заменить только компонент разбора аргументов, например на JSON/XML-парсер, не трогая фильтры, Pipeline и WAV I/O.

2. Как устроена архитектура
---------------------------

Основные компоненты:

    src/args/
        ArgsParser — разбор аргументов командной строки.

    src/filter/
        IFilter — общий интерфейс всех фильтров.

    src/filters/
        Конкретные фильтры: ampl, normalize, silence, timestretch, lowpass и т.д.

    src/filter_producers/
        FilterProducer — фабрики, которые создают нужный фильтр по FilterDescriptor.

    src/converter/
        CmdLineArgs2PipelineConverter — преобразует дескрипторы фильтров в Pipeline.

    src/pipeline/
        Pipeline — владеет фильтрами и применяет их по очереди.

    src/waveform/
        Waveform — модель звукового фрагмента в памяти.

    src/wav/
        WavReader / WavWriter — чтение и запись WAV.

    src/app/
        Application — координирует весь процесс.
        main.cpp — точка входа.

Почему так:
- каждый компонент имеет одну ответственность;
- фильтры не знают, как были заданы параметры;
- Pipeline не знает, откуда пришли фильтры;
- WAV I/O не знает про фильтры;
- Application собирает компоненты вместе.

3. Правило трех и правило пяти
------------------------------

Правило трех:
Если класс явно реализует хотя бы один из деструктора, copy constructor или copy assignment, нужно явно реализовать все три.

Правило пяти:
Если класс управляет ресурсами, нужно явно определить:
1. деструктор;
2. copy constructor;
3. copy assignment;
4. move constructor;
5. move assignment.

В проекте Pipeline раньше владел сырыми указателями IFilter*. Для такого класса правило пяти критично, потому что он освобождает ресурсы в деструкторе.

В текущей реализации:
- Pipeline запрещает копирование, чтобы не было двойного удаления фильтров.
- Pipeline разрешает перемещение, чтобы его можно было возвращать из функции.
- В move constructor переносится вектор указателей.
- В move assignment сначала удаляются старые фильтры, затем переносится владение.
- Деструктор удаляет все фильтры.

Почему копирование запрещено:
Если Pipeline копируется обычным copy constructor, оба объекта будут содержать одинаковые указатели. При удалении первого Pipeline фильтры будут удалены, а второй Pipeline будет хранить висячие указатели. Это UB. Поэтому копирование удалено.

Почему перемещение разрешено:
Функция create_pipeline() возвращает Pipeline по значению. Move-семантика позволяет эффективно передать владение фильтрами без лишних копий.

4. Rule of Zero
---------------

Некоторые классы не управляют ресурсами вручную, поэтому для них правило пяти не нужно. Например:
- ArgsParser хранит std::string и std::vector.
- Waveform хранит std::vector<int16_t>.
- FilterDescriptor хранит std::string и std::vector<std::string>.

Для них стандартные операции копирования/перемещения безопасны. Это пример Rule of Zero: если ресурсами управляют стандартные контейнеры, не нужно вручную писать деструкторы и операторы.

5. main.cpp
------------

Файл:

    tasks/sound_processor/src/app/main.cpp

Что делает:
- создает Application;
- вызывает app.configure();
- вызывает app.start(argc, argv);
- ловит std::exception и неизвестные исключения;
- возвращает понятные коды завершения.

Зачем:
ТЗ требует, чтобы main() был в модуле main.cpp и чтобы верхний уровень обрабатывал исключения.

Как объяснить:
main не должен содержать бизнес-логику. Он только запускает Application и обрабатывает ошибки, которые не были обработаны ниже.

6. application.cpp / application.h
----------------------------------

Файлы:

    tasks/sound_processor/src/app/application.cpp
    tasks/sound_processor/src/app/application.h

Что делает Application:
1. configure() регистрирует все FilterProducer в converter.
2. start() вызывает ArgsParser.
3. Если аргументы некорректны — выводит ошибку.
4. Если нет аргументов или запрошена справка — печатает help.
5. Создает Pipeline из дескрипторов фильтров.
6. Если нужен входной файл — читает WAV в Waveform.
7. Применяет Pipeline.
8. Если задан output — записывает результат.

Особенность генераторов:
Если первый фильтр — generator, входной файл не читается. Это соответствует ТЗ: генератор игнорирует входной сигнал.

Как объяснить:
Application — координатор. Он не реализует фильтры и не парсит параметры фильтров. Он только связывает компоненты.

7. args_parser.cpp / args_parser.h
----------------------------------

Файлы:

    tasks/sound_processor/src/args/args_parser.cpp
    tasks/sound_processor/src/args/args_parser.h

Что делает ArgsParser:
- принимает argc/argv;
- распознает -i, -o, -f, -h, --help;
- собирает FilterDescriptor;
- хранит параметры фильтра как строки;
- не проверяет семантику параметров фильтра.

Почему parser не проверяет параметры фильтров:
ТЗ прямо говорит: парсер не должен проверять корректность параметров конкретных фильтров, потому что это требует знания логики создания фильтров.

Пример:
Для `-f ampl 0.8` parser просто создает descriptor:
    name = "ampl"
    params = {"0.8"}

А проверку factor >= 0 делает AmplFilter или producer.

Что проверять parser может:
- дублирование -i/-o;
- отсутствие значения после флага;
- пустое имя файла;
- неизвестные флаги;
- ошибочные комбинации вроде `-f -i`.

8. filter.h
-----------

Файл:

    tasks/sound_processor/src/filter/filter.h

Что содержит:

    class IFilter {
    public:
        virtual ~IFilter() = default;
        virtual void apply(Waveform& waveform) = 0;
    };

Зачем:
Это общий интерфейс для всех фильтров. Благодаря ему Pipeline может хранить разные фильтры в одном контейнере:

    std::vector<IFilter*>

Почему деструктор виртуальный:
Pipeline удаляет фильтры через базовый указатель IFilter*. Если деструктор не виртуальный, будет удален только базовый подкласс, а ресурсы конкретного фильтра могут не освободиться. Это нарушает правило C.35 и может привести к утечкам.

9. filter_producers.h
---------------------

Файл:

    tasks/sound_processor/src/filter_producers/filter_producers.h

Что делает:
- преобразует текстовые параметры из FilterDescriptor в типизированные значения;
- проверяет количество аргументов;
- создает нужный фильтр через new;
- бросает runtime_error при неправильных параметрах.

Пример:

    make_ampl(fd)
        требует 1 аргумент;
        парсит double;
        создает AmplFilter(factor).

Зачем producer:
ТЗ требует избежать хардкода создания фильтров внутри parser или application. Producer — это фабрика, которая знает только один тип фильтра. Converter знает только map от имени фильтра к producer.

10. converter.cpp / converter.h
-------------------------------

Файлы:

    tasks/sound_processor/src/converter/converter.cpp
    tasks/sound_processor/src/converter/converter.h

Что делает:
- хранит map: имя фильтра -> FilterProducer;
- add_filter_producer() регистрирует producer;
- create_pipeline() проходит по дескрипторам;
- ищет producer;
- создает фильтр;
- добавляет фильтр в Pipeline.

Почему map:
ТЗ прямо рекомендует ассоциативный массив, чтобы не писать длинную цепочку if/else по именам фильтров.

Как объяснить:
Converter изолирует создание фильтров от parser. Если завтра появится новый фильтр, нужно:
1. написать класс фильтра;
2. написать producer;
3. зарегистрировать producer в Application::configure().

Никакие другие компоненты менять не нужно.

11. pipeline.cpp / pipeline.h
-----------------------------

Файлы:

    tasks/sound_processor/src/pipeline/pipeline.cpp
    tasks/sound_processor/src/pipeline/pipeline.h

Что делает Pipeline:
- хранит коллекцию IFilter*;
- add_filter() добавляет фильтр;
- apply() последовательно вызывает filter->apply(waveform);
- деструктор удаляет фильтры.

Почему Pipeline — отдельный компонент:
ТЗ требует цепочку фильтров как отдельную сущность. Это позволяет переиспользовать один и тот же Pipeline для разных Waveform.

Как объяснить:
Pipeline — это композиция фильтров. Он не знает, какие именно фильтры внутри, только что они реализуют IFilter.

12. waveform.cpp / waveform.h
-----------------------------

Файлы:

    tasks/sound_processor/src/waveform/waveform.cpp
    tasks/sound_processor/src/waveform/waveform.h

Что делает Waveform:
- хранит samples_ как std::vector<int16_t>;
- хранит sample_rate_, num_channels_, bits_per_sample_;
- умеет переводить секунды в samples;
- умеет переводить индекс сэмпла в секунды;
- проверяет out-of-bounds через vector::at();
- проверяет переполнение в seconds_to_samples().

Почему int16_t:
ТЗ требует 16-bit signed PCM. int16_t точно соответствует диапазону [-32768, 32767].

Почему vector:
- автоматически управляет памятью;
- безопаснее raw new/delete;
- поддерживает dynamic size;
- хорошо работает с resize/insert.

13. wav_io.cpp / wav_io.h / wav_headers.h
-----------------------------------------

Файлы:

    tasks/sound_processor/src/wav/wav_io.cpp
    tasks/sound_processor/src/wav/wav_io.h
    tasks/sound_processor/src/wav/wav_headers.h

Что делает WavReader:
- открывает файл через std::ifstream в binary mode;
- читает RIFF header;
- проверяет "RIFF" и "WAVE";
- проверяет размер RIFF chunk;
- читает fmt chunk;
- проверяет PCM, mono, 44100 Hz, 16-bit;
- проверяет byte_rate и block_align;
- ищет data chunk;
- проверяет, что data chunk не больше реального файла;
- читает samples в vector<int16_t>.

Что делает WavWriter:
- открывает файл через std::ofstream в binary mode;
- проверяет, что Waveform mono 44100 Hz 16-bit;
- пишет RIFF/fmt/data headers;
- пишет samples.

Почему std::fstream:
ТЗ требует использовать стандартные файловые потоки, а не C FILE*.

Почему packed structs:
WAV header — бинарный формат с фиксированными полями. Packed structs позволяют читать и писать заголовки блоками, уменьшая количество IO-операций.

14. filters/ampl_filter.h
-------------------------

Что делает:
- умножает каждый sample на factor;
- factor должен быть finite и >= 0;
- результат clamp в диапазон int16_t.

Формула:

    s'[i] = clamp(s[i] * factor)

15. filters/normalize_filter.h
------------------------------

Что делает:
- находит текущий максимальный модуль sample;
- если сигнал пустой или тихий — ничего не делает;
- вычисляет scale = peak * 32767 / currentPeak;
- масштабирует все samples.

Почему currentPeak хранится в int32_t:
abs(INT16_MIN) = 32768, это не помещается в int16_t. int32_t предотвращает переполнение.

16. filters/silence_filter.h
----------------------------

Что делает:
- вставляет нулевые samples в диапазон [start, end];
- start/end могут быть в sec или ms;
- диапазон inclusive по ТЗ;
- если вставка за концом сигнала — дописывает нули в конец.

Почему +1:
ТЗ говорит, что end включительно. Поэтому если start и end дают один и тот же индекс, вставляется один нулевой sample.

17. filters/timestretch_filter.h
--------------------------------

Что делает:
- меняет длительность сигнала в factor раз;
- factor > 0;
- новая длина = round(oldSize * factor);
- для каждого нового sample вычисляется позиция pos = i / factor;
- используется линейная интерполяция между соседними samples.

Формула:

    value = s[l] * (1 - frac) + s[l + 1] * frac

Если правого соседа нет, берется s[l].

Почему это лучше простого дублирования:
В ТЗ есть подробное описание линейной интерполяции. Простое дублирование samples было бы менее точным.

18. filters/lowpass_filter.h
----------------------------

Что делает:
- сглаживает сигнал скользящим средним;
- window_size должен быть нечетным и >= 1;
- на краях индексы clampятся к границам сигнала.

19. filters/highpass_filter.h
-----------------------------

Что делает:
- вычитает из сигнала moving average;
- тем самым убирает низкочастотную/постоянную составляющую;
- использует тот же принцип окна, что lowpass.

20. filters/bandpass_filter.h
-----------------------------

Что делает:
- сначала lowpass с большим/низким окном;
- затем вычитает еще более сглаженную версию;
- оставляет полосу частот между двумя уровнями сглаживания.

21. filters/reject_filter.h
---------------------------

Что делает:
- вычисляет bandpass-сигнал;
- вычитает его из исходного сигнала;
- таким образом удаляет среднюю полосу.

Почему bandpass и reject не дублируют друг друга:
- bandpass оставляет среднюю полосу;
- reject удаляет среднюю полосу и оставляет остальное.

22. filters/generator_filters.h
-------------------------------

Что делает:
- AbstractGeneratorFilter задает общий apply() для генераторов;
- SinGeneratorFilter создает синус;
- AmGeneratorFilter создает амплитудно-модулированный сигнал;
- FmGeneratorFilter создает частотно-модулированный сигнал.

Общее:
- генераторы игнорируют входной Waveform;
- выставляют стандартные метаданные 44100 Hz, mono, 16-bit;
- заполняют samples заново.

Формулы:
- sin: 32767 * sin(2 * pi * frequency * t)
- AM: amplitude * 32767 * envelope * carrier
- FM: amplitude * 32767 * sin(phase)

23. filters/mute_filter.h
-------------------------

Дополнительный фильтр.
Зануляет samples в диапазоне [start_sec, end_sec).

24. filters/mix_filter.h
------------------------

Дополнительный фильтр.
Накладывает второй WAV поверх основного:
- читает дополнительный Waveform;
- сдвигает его на start_sec;
- суммирует samples;
- clamp в int16_t.

29. Главные принципы
--------------------

SOLID/архитектура:
- Single Responsibility: каждый компонент отвечает за одну задачу.
- Open/Closed: новый фильтр добавляется без переделки parser/pipeline.
- Dependency inversion через IFilter: Pipeline зависит от абстракции, а не от конкретных классов.

C++:
- Rule of Zero там, где ресурсы хранятся в std::vector/std::string.
- Rule of Five там, где класс владеет ресурсами.
- Виртуальный деструктор в базовом полиморфном классе.
- RAII через стандартные контейнеры и потоки.
- Отсутствие FILE*, fread/fwrite.
- Проверка исключений на верхнем уровне.

30. Если спросят про слабые места
---------------------------------

Можно честно сказать:

- WAV reader ориентирован на PCM mono 44100 Hz 16-bit, как требует ТЗ.
- Частотные фильтры — простые оконные реализации, не FFT/IIR.
- Generator FM использует упрощенную формулу фазы из ТЗ.
- Mix/mute добавлены как дополнительные фильтры сверх обязательного набора.

Это нормально, потому что ТЗ требует именно простую реализацию фильтров и фиксированный WAV-формат.
