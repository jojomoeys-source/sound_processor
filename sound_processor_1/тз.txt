Введение

Целью данной работы является выработка у студентов, обучающихся на курсе «Основы программирования на C++», навыков построения на языке программирования C++ сложных программных проектов индустриального уровня, включающих изучения основ цифровой обработки звуковых сигналов.
Разрабатываемый проект представляет собой консольное (CLI) приложение, выполняющее роль цифрового обработчика звукового файла в формате WAV, поданного ему на вход. Полученный файл пропускается через цепочку фильтров (пайплайн), набор и характеристики которых также передаются приложению в виде параметров командной строки. Фильтры, входящие в пайплайн, выполняют последовательное преобразований звуковых данных исходного файла. Примеры преобразований, реализуемых фильтрами:

изменение амплитудных характеристик звуковой волны (параметрическое изменение громкости звука, вставка фрагментов «тишины» — паузы, нормализация звука);
простые изменения частотно-временных характеристик звуковой волны (сжатие, растяжения звукового фрагмента);
генерация простейших параметрических звуковых сигналов (синусоид);
генерация амплитудно-модулированного параметрического звукового сигнала;
генерация частотно-модулированного параметрического звукового сигнала;
применения простых реализация параметрических фильтров нижний частот (ФНЧ), верхних частот (ФВЧ), полосовых и режекторных фильтров.

Разрабатываемое приложение имеет компонентную архитектуру, которая позволяет выполнять модификацию отдельных функциональных фрагментов приложения путем замены соответствующих компонент с фиксированным API без необходимости инвазивного внесения изменений в множественные участки кода всего приложения. Требования и ограничения, накладываемые на архитектуру приложения, предписываются настоящим заданием и обязательны к соблюдению.
Для реализации проекта студент должен владеть основами языка C++, функциональной декомпозиции, объектно-ориентированной разработки и проектирования.
Описание разрабатываемого приложения

Кодовое название разрабатываемого приложения: «Sound Processor» (название таргетов, директорий, основных компонентов, пакетов и др: sound_processor).
Разрабатываемое приложение представляет собой консольное приложение с интерфейсом командной строки (CLI). Входные параметры, определяющие имена входных и выходных файлов, названия применяемых фильтров и их параметры, передаются на вход приложения через механизм аргументов командной строки.
Входными данными приложения являются характеристики звукового сигнала, представленные в закодированном виде — WAV-файле с импульсно-кодовой модуляцией (ИКМ). Результатом работы приложения (выходными данными) в общем случае является модифицированный звуковой файл, также при необходимости определенные данные могут выводиться в стандартный поток вывода stdout или стандартный поток ошибок stderr. Интерактивное взаимодействие приложения с пользователем путем запроса необходимых данных через стандартный поток ввода stdin не предусмотрен. Основная часть приложения, выполняющего преобразование звуковых данных (применение фильтров, объединенных в единую цепочку — пайплайн), должна быть инвариантна относительно способа описания и ввода (поступления) в приложения входных данных. Это означает, что расширение приложения для ввода данных не из командной строки, а из другого источника (например, из xml, json-файла, из сетевого сокета и др.) должно быть принципиально изолировано от процесса применения фильтров и реализовано в независимом формат-специфичном компоненте с заданным интерфейсом. Таким образом, потенциальный переход от ввода параметров из командной строки к условному json-файлу должен сводиться к разработке нового специализированного компонента разбора структуры этого json-файла и порождению на его основе цепочки фильтров. Этот компонент заменяет собой разрабатываемый в данном проекте компонент разбора параметров командной строки без влияния на остальную структуру проекта.

Описание параметров командной строки

В данном проекте необходимо реализовать только парсер аргументов командной строки с учетом ограничений на архитектуру, указанных в других разделах.
Структура параметров командной строки определяется следующим образом:

{имя_приложения} [-i путь_к_входному_файлу] [-o путь_к_выходному_файлу] 
[-f {название_фильтра1} [параметр1_фильтра1] [параметр2_фильтра1]...] 
[-f {название_фильтра2} [параметр1_фильтра2] [параметр2_фильтра2]...] ...


При запуске без переданных аргументов программы выводит справку по работе.
Пример (1) вызова приложения:

sound_processor -i input.wav -o output.wav -f ampl 0.8 -f timestretch 2


В данном примере вызывается приложение звуковой процессор через имя исполнимого файла sound_processor (в вариантах для ОС Linux и MacOS; для ОС Windows название исполнимого файла будет sound_processor.exe, однако при вызове его из командной строки расширение можно опустить, поэтому целиком команда вызова будет совпадать с таковой для Linux и MacOS), читает входной файл input.wav (что указывается ключом -i), применяет к нему последовательно фильтры (задаются с помощью ключа -f):


ampl: абсолютное изменение амплитуды звукового сигнала (значения отсчетов) устанавливается в 80 % от исходного;

timestretch: существующий звуковой сигнала растягивается по времени в 2 раза (выборки дублируются);

После этого преобразованный звуковой сигнал записывается в выходной файл output.wav (что указывается ключом -o).
Пример (2) вызова приложения:

sound_processor -o output.wav -f generator sin 440 2000 -f ampl 1.1


В данном примере входной звуковой файл не читается (ключ  -i отсутствует), вместо этого первым исполняемым фильтром является фильтр generator, который осуществляет генерацию (то есть создание) эталонного звукового сигнала с параметрами, указанными далее:


sin: стандартный синусоидальный сигнал

440: частотой 440 Гц

2000: длительностью 2000мс (2 сек).

К сгенерированному сигналу затем применяются следующие фильтры, в данном примере — фильтр ampl, описанный выше, после чего результат сохраняется в выходной файл output.wav (ключ -o).

ℹ️ Замечание: для унификации программной реализации все фильтры разрабатываются таким образом, чтобы получать единообразный вход (звуковой сигнал) и формировать единообразный выход (также звуковой сигнал). Фильтры семейства generator (генераторы) работают аналогично всем остальным фильтрам, однако они игнорируют входной звуковой сигнал (то есть толерантны к NULL-объекту) и формируют выходной сигнал исключительно на основе задаваемых параметров (см. рис. ниже):

Если не вход фильтру-генератору подается непустой звуковой сигнал, он игнорируется фильтром. Таким образом, использование опции -i совместно с фильтром-генератором формально не должен рассматриваться как ошибочный вход: сигнал из входного файл просто игнорируется после применения фильтра-генератора.

Если при запуске приложения не передан ни один фильтр для применения, тогда исходный сигнал записывается в выходной файл в неизменном виде. Если в приложение не передан входной файл и не применен ни один фильтр-генератор, в результирующий файл записывается пустой звуковой сигнал, при этом стандартные заголовки RIFF-контейнера и WAVE-чанка (см. ниже) записываются корректно. Если в приложение не передан выходной файл, сформированный сигнал не записывается никуда.
Описание формата звукового файла WAV

Исходные данные для работы приложения представлены в формате WAV — формат файла-контейнера для хранения оцифрованного аудиопотока (wiki, radioprog). Формат основан на контейнерном подходе RIFF, его содержимое состоит из секций, следующих друг за другом последовательно или условно вкладываемые друг в друга.
Формат контейнеров WAV допускает многочисленные варианты представления звуковых данных. Для упрощения в данной работе приняты договоренности относительно формата, на которые можно ориентироваться при разработке компонентов обработки файлов (чтение, запись):

формат аудио: импульсно-кодовая модуляция (PCM)
число каналов: 1
частота дискретизации: 44100 Гц
число бит на выборку: 16, знаковые целые


ℹ️ Замечание: для исследования и работы с «сырыми» двоичными файлами рекомендуется использовать HEX-редактор (бинарный редактор): HxD Editor, ImHex, Hex Editor Neo, 010 Editor, GHex, Bless, Hex Fiend и др. Можно также воспользоваться плагинами-расширениями для популярных IDE: Hex Editor для VSCode, Hex Editor или BinEd для CLion.

Несмотря на то, что формат WAV-файла допускает появление разных фрагментов (чанков, chunk), в том числе повторное появление чанков с данными, в данной работе можно ориентировать на ограничение, что предлагаемые образцы звуковых файлов будут состоять из трех основных чанков (см. хорошее описание на radioprog):



Номер, название
Смещение отн. начала файла, hex

1. RIFF chunk

размер файла минус 8 байт
Главный чанк, определяющий структуру контейнера RIFF. Все остальные чанки являются (условно) вложенными в него, поэтому его размер определяется как размер всего файла за вычетом 8 байт, определяющих заголовок этого чанка (RIFFб 4 байта) и размер оставшейся части файла (тела чанка, 4 байта). Описание содержимого RIFF-контейнера занимает следующие 4 байта по смещению 8h и равно WAVE. Таким образом, общий размер заголовочной части RIFF-чанка — 12 байт.


2. format chunk
Ch
24 байт
Определяет структуру кодирования звуковых данных (см. ограничения сверху и пример снизу).


3. data chunk
24h
размер файла минус 36 байт
Содержит выборки звукового сигнала. Сами выборки начинаются по смещению 24h + 8h (размер заголовка чанка data) = 2Ch. Выборки единственного моно-канала следуют подряд, одна за другой, по 16 бит (2 байта) на выборку, число выборок определяется продолжительностью звукового файла, сек * частоту дискретизации 44100 выб/сек. Общее число байт под выборки определяется как число выборок * 2 (16 бит на выборку).

Архитектура приложения и реализация

Приложение строится по компонентному принципу. Данный раздел включает описание обязательных и рекомендуемых компонент, а также требования и рекомендации по программному интерфейсу (API) компонент и их взаимодействию.
Обязательные компоненты

Приложение обязательно должно включать следующие компоненты, разработанные в виде набора одного или нескольких взаимодействующих классов, интерфейсов и, при необходимости, — свободных методов. Каждый компонент должен быть самодостаточен, изолирован, тестируем, переиспользуем, заменяем. Реализация любого компонента должна быть выполнена таким образом, чтобы внесение в нее любых изменений или полная ее замена никаким образом не отражалась на функционировании остальных частей приложения и не требовало их корректировки. Это означает, что главным и наиболее критическим является правильная разработка интерфейсных частей компонент, изменение которые в дальнейшем либо не предусматривает, либо выполняется редко.
Список обязательных компонентов:

парсер аргументов командной строки;
звуковой фрагмент (waveform);
компонент(ы) чтения/записи звуковой информации из/в WAV-файл;
интерфейс фильтра;
фильтры;
цепочка применяемых фильтров (пайплайн);
конвертер, преобразующий распарсенные аргументы командной строки в пайплайн.

Также рекомендуется создать компонент Application, организующий работу остальных компонент, в виде отдельного класса. Реализация компонента в виде функции (разновидность main()) возможна, но менее предпочтительна.
Далее приводятся комментарии относительно отдельных компонентов.
Компонент «Парсер аргументов командной строки» (ПАКС)

Назначение: принимает на вход стандартизированные аргументы командной строки в виде пары объектов (int argc, char* argv[]), выполняет разбор аргументов, проверяет их первичную корректность (в части допустимых флагов и их комбинаций).

⚠️ Парсер не должен проверять корректность параметров конкретных фильтров, т.к. это требует включения логики создания фильтров, что выходит за зоны ответственности ПАКС.

Парсер должен выделять и предлагать в дальнейшем через API доступ к следующим параметрам:

имя входного и выходного файла;
список дескрипторов фильтров.

Дескриптор фильтра — множество выделенных корректно переданных токенов, начинающихся с флага -f и разделенных пробелом. Первым выделенным токеном является название фильтра, которое в рамках объекта-дескриптора должно быть доступно отдельно от его параметра. Остальные токены представляют собой параметры фильтра. Так как все параметры в АКС представляют собой текстовые токены, а их интерпретацию в виде чисел и других объектов могут дать только компоненты, выполняющие построение объекта-фильтра на основе этих токенов, рекомендуется хранить эти параметры как есть в виде текстовых строк. При этом, учитывая глобальную доступность строк в массиве char* argv[], допускается хранить параметры-токены в виде указателей char* без преобразования в более комплексные типы строк вида std::string.
Рассмотрим модифицированный пример (1) выше:

sound_processor -i input.wav -o output.wav -f ampl 0.8 -f silence sec 0.2 0.4


В этом примере фильтр с названием silence вставляет «тишину» в момент времени между 0,2 сек и 0,4 сек и имеет три параметра: sec — единицы времени (секунды), 0,2 — точка начала и 0,4 — точка окончания тишины (включительно).
Здесь при вызове функции:

значение argc == 13, а массив argv содержит указатели на следующие токены-строки:

{ "sound_processor "-i" "input.wav" "-o" "output.wav" "-f" "ampl" "0.8" "-f" "silence" "sec" "0.2" "0.4" }

Компонент «Звуковой фрагмент (waveform)»

Назначение: представляет собой объектную модель звукового сигнала, загруженного из файла в оперативную память для дальнейшей манипуляции с ним.
Рекомендуемая реализация: класс, включающий метаинформацию о звуковом фрагменте и набор отсчетов, загруженные в двоичной форме. Метаинформация по большей части отражает содержимое полей чанка format звукового файла WAV. С учетом фиксированных значений большинства полей (битрейт, частота дискретизации, моно-канал) основным вариантивным атрибутом данного объекта является число выборок, которое линейным образом связано с длительностью звукового фрагмента. Рекомендуется разработать набор вспомогательных методов (helper-ов) для «удобной» работы с этим параметром: увеличение/уменьшение длительности, пересчет между различными единицами измерения (выборки, секунды, милисекунды) и др.
Сами выборки рекомендуется хранить с помощью любого стандартизированного контейнера, эффективно управляющего динамической памятью, например:

std::vector<int16_t>


В данном случае для представления одной выборки достаточно использовать 16-битное знаковое целое int16_t.
Для выполнения математики над индивидуальными выборками следует осуществить преобразование в числа с плавающей точкой (например, double), однако хранить все выборки в виде таких чисел не представляется обоснованным — если не возникает необходимость реализации какого фильтра, которому нужен многократный случайный доступ к различным участкам звукового фрагмента, что сделало бы постоянное преобразование менее эффективным, чем хранение однократно преобразованных данных.
Далее, для определенности, будем ссылаться на такой компонент как на владеющий1 класс Waverform.
Компонент(ы) чтения/записи звуковой информации из/в WAV-файл


⚠️ Требование: для выполнения операций чтения/записи следует использовать файловые потоки стандартной библиотеки, а не низкоуровневые C-функции, работающие с файловыми дескрипторами FILE*.

Назначение: компонент(ы) инкапсулируют и реализуют операции для работы с потоками, открытыми в двоичном режиме, для чтения звуковых данных из файла формата WAV и соответствующей их записи.
Могут быть реализованы в виде единого класса, выполняющего как чтение, так и запись, тогда для входного и выходного файла создается по одному экземпляру такого файла. Допускается определить отдельный класс для чтения и отдельный для записи; при этом, общий функционал может быть вынесен в базовый класс.
Для чтения/записи стандартизированных структур чанков RIFF-файла рекомендуется определять соответствующие упакованные C-структуры, например:

struct __attribute__((packed)) RiffHeader
{
    uint32_t sign;
    uint32_t size;
    uint32_t waveid;
};

struct __attribute__((packed)) FmtHeader
{
    uint32_t chunkId;
    uint32_t chunkSize;
    uint16_t wFormatTag;
    ...
    ...
};



При выполнении чтения/записи следует стремиться минимизировать число IO-операций в силу их большой временной стоимости, поэтому фиксированные структуры следует вычитать (записывать) за одну операцию, а не каждое поле индивидуально:

std::istream strm = .......;
RiffHeader riffHdr;
strm.read(reinterpret_cast<char*>(&riffHdr), sizeof(riffHdr));


В результате чтения корректного WAV-файла компонент возвращает сформированный объект «Звуковой фрагмент (waveform)». При выполнении записи, объект типа «Звуковой фрагмент (waveform)» является обязательным параметром. В результате записи должен быть сформирован новый корректный WAV-файл, удовлетворяющий требованиям, указанным в разделе «Описание формата звукового файла WAV».
Компонент «Интерфейс фильтра»

Реализация алгоритма некоторого фильтра сильно зависит от конкретной задачи, которая им решается. Разные фильтры могут иметь разные параметры — по числу и по типам, а значит свести применения фильтра к вызову некоторого унифицированного метода становится затруднительным. Для этого пришлось бы передавать вариативное множество разнотиповых параметров в виде массива некоторого вариантного типа данных, при этом остается проблема правильного преобразования этих параметров из текстовых представлений токеном АКС в соответствующие типизированные значения:

applyFilter(Waveform* sound, Variant param[]);


Другим — и более подходящим для данной задачи подходом — будет применение полиморфной системы типов фильтров, реализующих базовый интерфейс абстрактного фильтра, который умеет применять себя (вернее — конкретную реализацию некоторого своего потомка) к переданному звуковому фрагменту — путем вызова виртуального метода apply(). Также, параметры конкретного фильтра теперь нет необходимости передавать в метод apply(); вместо этого они становятся атрибутами объекта класса соответствующего фильтра и могут инициализировать этот объект при конструировании (параметрами конструктора) или после — путем вызова соответствующих setter-ов.
Возможный вид интерфейсного базового (абстрактного) класса фильтра может быть таким:

class IFilter {
public:
    virtual ~IFilter {}
    virtual State apply(Waveform* sound) = 0;
};


Здесь класс IFilter включает объявления публичного виртуального деструктора в соответствии с правилом C.35 — в дальнейшем это понадобится нам для реализации пайплайна как владеющего объекта  коллекцией фильтров через полиморфный базовый указатель. Также объявляется чисто виртуальная функция apply(), получающая параметром звуковой объект для применения к нему фильтра и возвращающая некоторый объект статуса (State м.б. реализован в виде enum по аналогии с парсером АКС, а также метод может возвращать bool, либо иное).

Компонент «Цепочка применяемых фильтров (пайплайн)»

Цепочка фильтров, последовательно применяемая к некоторому фрагменту звукового файла, должна быть реализована в виде индивидуальной сущности «пайплайн». Мотивация: возможность переиспользования настроенной последовательности фильтров для более чем одного звукового файла 2.
Заготовка класса для сущности «пайплайн» может выглядеть следующим образом:

class Pipeline {
public:
    // implementing the Rule of 5 here...
    ~Pipeline() 
    {
        for(Filter* filter : _filters)
            delete filter;
    }
    
    State apply(Waveform* sound);
public:
    Filter* addFilter(Filter* filter);
    size_t getFilterNumbers() const;
    Filter* operator[](size_t i) const;
    
    // ... other methods for enumerating filters...
protected:
    std::vector<Filter*> _filters;
};


Важнейшими методами в предложенной заготовки являются деструктор ~Pipeline() и метод apply(), применяющий сконструированный пайплайн к некоторому переданному звуковому фрагменту.
Подход к представлению фильтров через полиморфные классы позволяет решить проблему владения коллекцией разнородных объектов фильтров путем создания коллекции (в данном примере — вектора) полиморфных базовых указателей. Пополнение такой коллекции просто:

_filters.push_back(new SinGenFilter(0.6, 440, 2000));
_filters.push_back(new AmplFilter(0.8));


В целях изоляции деталей реализации реальное добавление фильтров должно осуществляться через вспомогательные методы:

pipeline.addFilter(new SinGenFilter(0.6, 440, 2000));
pipeline.addFilter(new AmplFilter(0.8));


Удаление создаваемыми в динамической памяти объектов фильтров осуществляется в деструкторе класса пайплайна: в результате удаления конкретного объекта фильтра через полиморфный базовый указатель будет вызван правильный деструктор именно того класса фильтра, объект которого подлежит под этим указателем. Именно по этой причине в соответствии с правилом C.35 деструкторы у класса IFilter и всех его потомков объявлены виртуальными.
Альтернативным вариантом является оборачивание объектов фильтров в умные указатели и хранение в векторе именно их: тогда удаление фильтров из кучи будет инициировано деструктором вектора при удалении объекта пайплайна.
Студентам необходимо предусмотреть ситуации, при которых возникает необходимость в копировании/перемещении объекта пайплайна. Для этого либо необходимо реализации методов копирования/перемещении в соответствии с правилом «пятерки», либо запрет такой семантики и передача объекта пайплайна между функциями с применением механизмов (например, ссылок), исключающих необходимости копирования.
Реализация метода apply() также не представляет затруднений и аналогична предложенной реализации деструктора: необходимо перебрать все включенные в пайплайн фильтры и для каждого из них вызвать виртуальный метод apply(), анализируя возвращаемое им значение (или предусмотрев возможность генерации исключений) и управляя возможностью дальнейшего применения фильтров в случае сбоя одного из них на очередном шаге.
Компонент «Конвертер, преобразующий распарсенные аргументы командной строки в пайплайн»

Одной из важнейший целей, которые необходимо достичь при реализации проекта, — принципиальное избежание хардкодинга создания/применения фильтров одновременно с разбором аргументов командной строки. Такой подход, хотя и является первоочевидным и прямолинейным, лишает разработчика возможности гибкого управления добавлением новых типов фильтров и чреват сложнонаходимыми ошибками. Например:

for(int i = 0; i < argc; ++i)
{
    if (argv[i] == "ampl")
    {
        double ampl = atod(argv[i + 1]);
        // if... если преобразование не получилось
        Filter* amplFilter = new AmplFilter(ampl);
        amplFilter->apply(sound);
        // или
        applyAmplFilter(sound, ampl);
    }
    else if (argv[i] == "gener")
    {
    	...
    }
    else if (argv[i] == "timestretch")
        ....
}



⚠️ Представленный выше код условный и дан исключительно для иллюстрации исключительно плохого подхода — не надо так делать.

Вместо этого предлагается разработать компонент-посредник, который, с одной стороны, будет все знать о формате разобранных парсером аргументах командной строки, а, с другой, — о типах фильтрах, которые надо на основе этих аргументов (после попытки их интепретации) создавать. Такой посредник, среди прочего, изолирует между собой АКС, как только один из возможных вариантов ввода параметров в приложения, и фильтров/пайплана, которые вообще ничего не должны «знать» о том, как именно параметры (фильтров) в приложение были введены. Рисунок ниже иллюстрирует предлагаемый подход:

Здесь в роли компонента-посредника выступает класс CmdLineArgs2PipelineConverter, который находится в отношении агрегации с коллекцией дескрипторов фильтров, формируемой парсером АКС (см. соответствующий раздел) — изображен в левой части рисунка, и пайплайном (как коллекцией фильтров соответствующих типов, владение которыми осуществляется через полиморфные базовые указатели) — изображен в правой части рисунка. В нижней части рисунка серым цветом изображены опциональные аналогичные конвертеры, выполняющие преобразование структуры фильтров, заданных с помощью XML- и JSON-файлов соответственно.

В данной работе конвертеры для XML- и JSON-файлов реализовывать не требуется.

Такая схема позволяет пайплайну и фильтрам быть определяемыми инвариантно исходного источника, в котором они были описаны для конкретного случая.
Первоначальная заготовка для класса CmdLineArgs2PipelineConverter может выглядеть следующим образом:

class CmdLineArgs2PipelineConverter {
public:
    Pipeline createPipeline(const std::vector& FilterDescriptor);
    // ...
};


Здесь создаваемый объект Pipeline возвращается по значению, для эффективной реализации чего необходима реализации правила «пятерки», о чем было сказано выше.
Следующий вопрос, который необходимо решить: как CmdLineArgs2PipelineConverter создает объекты фильтры (перед помещением их в пайплайн), опираясь на очередной данный дескриптор фильтра? Каждый дескриптор состоит из названия фильтра и набора текстовых токенов, число которых определяется разновидностью фильтра и этим же определяется способ их интерпретации (в виде целых чисел, с плавающей точкой и т.д.).
Если включить в реализации метода createPipeline() цепочку if...else if..., которая анализирует название фильтра в дескрипторе, мы придем к проблеме, подробно описанной в начале раздела. Помимо сложности и ошибкоемкости такого подхода, это еще и лишает разработчика возможности оперативного добавления новых разновидностей фильтров: по меньше мере, для этого пришлось встраивать в цепочку еще одну ветвь else if..., что требует перекомпиляции приложения и делает принципиально невозможным динамическое пополнение системы фильтров, например, путем подключения динамических плагинов при запуске приложения.
Решением этой проблемы является применение ассоциативного массива, который ставил бы соответствие между названием разновидности фильтра (“ampl”, “timestretch”, “gener”...) и некоторым механизмом FilterProducer, который порождал бы фильтр необходимого типа на основе параметров для этого фильтра, хранимые в дескрипторе. В роли такого ассоциативного массива можно использовать стандартные контейнеры STL, например:

std::map<const char*, FilterProducer> _producers;


Реализации класса CmdLineArgs2PipelineConverter с учетом предложенного принимает вид:

// file cmd_line_args_2_pipeline_converter.h

class CmdLineArgs2PipelineConverter {
public:
    Pipeline createPipeline(const std::vector& FilterDescriptor);
    // ...
    addFilterProducer(const char* filterName, FilterProducer);
    FilterProducer getFilterProducer(const char* filterName);
private:
    std::map<const char*, FilterProducer> _producers;
};

// file cmd_line_args_2_pipeline_converter.cpp

FilterProducer CmdLineArgs2PipelineConverter::getFilterProducer(const char* filterName)
{
    auto it = _producers.find(filterName);
    if (it == _producers.end())
        return nullptr;
    
    return it->second;
}


В предложенном коде пополнение коллекции продюсеров осуществляется вызовом вспомогательного метода addFilterProducer(). На рисунке выше коллекция продюсеров изображена под компонентом CmdLineArgs2PipelineConverter фиолетовым цветом. Подразумевается, что пополнение коллекции продюсером осуществляется на этапе инициализации-конфигурирования приложения, еще до разбора аргументов к.с., т.к. эта этап относится именно к конфигурированию, а не к решению конкретной задачи по применению фильтров к изображению. В случае разработки вспомогательного класса Application рекомендуется выносить инициализацию продюсеров в отдельный метод: например, в разделе «Стартовый код: функция main()» (см. ниже) в этой роли выступает вызов app.configure().
Последний вопрос, который необходимо решить в контексте этой подзадачи: как реализовать сущность FilterProducer так, чтобы она удовлетворяла следующий требованиям:

для любых разновидностей фильтров реализаций FilterProducer должна быть типоунифицированной, т.к. только в этом случае FilterProducer можно использовать в качестве типа значения ассоциативного массива;
реализация FilterProducer должна принимать на вход дескриптор фильтра и порождать фильтр соответствующего (названию в дескрипторе) типа, возвращаемого в виде унифицированного полиморфного базового указателя IFilter*; на этом этапе теряется специфичность типа фильтра, однако, принимая во внимание, что в пайплайне все фильтры хранятся и используются через коллекцию IFilter*, это не является проблемой;
в случае, если для переданного в метод getFilterProducer() названия фильтра не находится подходящий FilterProducer, необходимо уведомить об этом вызывающую функцию, по возможности избегая возврата громоздких объектов и генерации исключительных ситуаций.

На основе анализа приведенных требований следует, что для найденного для заданного имени фильтра типа продюсера обращение к нему для создания фильтра может выглядеть следующим образом:

FilterDescriptor fd {.name = "ampl", .params = {0.8}};
FilterProducer prod = getFilterProducer(fd.name);
if (prod)
{
    IFilter* filter = prod(fd);
    if (filter)				// if !filter here, there are som trouble with params
    {    					// or so on
        pipeline.addFilter(filter);
    }
}


Из приведенного кода становится очевидным, что в роли FilterProducer должен выступать любой функциональный объект, как то: указатель на функцию, функтор, std::function и т.д. Наиболее простым и полностью покрывающим все потребности является указатель на функцию. Действительно, достаточно определить тип для таких объектов одним из следующих подходов:

using FilterProducer = Filter* (*) (const FilterDescriptor&);  // modern approach
// or, alternatively:
typedef Filter* (*FilterProducer) (const FilterDescriptor&);  // classical approach


и приведенный выше код становится полностью гармонизированным и выполняющим поставленную задачу с учетом введенных ограничений.

⚠️ Важные замечания. Функционал по нахождению, применению FilterProducer-ов, проверки ненулевых указателей самого FilterProducer и создаваемого им фильтра необходимо изолировать на уровне соответствующих методов внутри класса CmdLineArgs2PipelineConverter, так как вынесение любой части функционала за пределы этого класса (как это сделано в примерах выше в иллюстративных целях) приводит к антипаттерну cross-cutting concern.

}

Стартовый код: функция main()

Стартовая функция main() программы, либо функция, непосредственно вызываемая ею для решения основной задачи, должна содержать блок проверки исключительных ситуаций, которые не были обработаны ни на одном уровне ниже, например так:

int main(int argc, char* argv[])
{
    Application app;
    int res = 0;
    try {
        app.configure();
        res = app.start(argc, argv);
    }
    catch(std::exception& e)
    {
        std::cerr << "An exception handled: " << e.what();
        return -1;	// a proper named const must appear here
    }
    catch(...)
    {
        std::cerr << "An unknown exception";
        return -2;	// a proper named const must appear here
    }    
    ...
    return res;
}

Математика фильтров реализуется студентами самостоятельно. Использование любых сторонних библиотек и специализированных библиотек для обработки звIFilterука (например, JUCE) запрещено.
Все исключительные ситуации должны корректно обрабатываться с выводом понятного пользователю сообщения об ошибке.
Никакие сценарии, включая использование файлов с форматом, не соответствующим спецификации, не должны приводить к падению программы.
При реализации проекта необходимо избегать появления классических антипаттернов проектирования, например, cross-cutting concern, non OOP-class и др. Реализация должна удовлетворять общекурсовым требованиям к code style и code convetions.
Структура проекта

Проект реализуется в рамках общего фреймворка для выполнения индивидуальных работ. Каталог sound_processor, включаемый в общий список задач, состоит из двух таргетов.


sound_processor — собственно, приложение, которое разрабатывается с нуля, включает функцию main(), которая в обязательном порядке помещается в модуль main.cpp. Разбиение остальных компонентов приложения по модулям не регламентируется (рекомендуется для каждого независимого компонента или набора связанных компонентов — например, стандартных фильтров, — создавать свой логический модуль, состояющий из пары «заголовочный файл + модуль трансляции»).
